# メール整理ルール

1. 未読メールの一覧化・トリアージ・既読化提案をするときは、このファイルを参照する。
2. メールの内容は本人の指示無しに読まないこと


## カテゴライズ

未読メールなどを一覧化するときは、以下のカテゴリに分けて表示する。

1. **要対応・人からのメール** — 個人からの返信・依頼など、自分のアクションが要りそうなもの
2. **Backlog 課題** — Backlog からの課題通知（件名が `[PROJECT-番号]` 形式のもの）
3. **システム通知・自動配信** — Cisco/ServiceNow/Backlogレポート/ToDoリスト/Google Chat 通知など機械配信のもの
4. **情報共有** — JINS情報発信会議など、社内のお知らせ・周知系
5. **アラート系** — USEN STB アラートなど、障害・復旧の自動通知

- 並び順は上記のとおり、要対応を上・自動通知系を下にする（優先度順）。
- 各カテゴリ内は新しい順。
- 件数で集約せず、**必ず1通ごとに件名を一覧表示する**
  （USEN STB アラートや Google Chat 通知のような同種の大量メールも省略しない）。

## 既読候補

**スキップしてよいのは以下に明示したものだけ。それ以外はすべて確認対象として残す**
（ホワイトリスト方式）。判断に迷うものもスキップせず残すこと。

該当メールは中身を確認せず既読化してよい。ただし一覧表示するときは、
件数だけで済ませず**1通ごとに件名を一覧表示する**こと。

- **情報共有**（JINS情報発信会議などの周知系）
- **USEN STB アラート**（usen-stb-alert、障害・復旧通知）
- **Backlog の週次マイレポート**（`no-reply@backlog.jp` の「今週のBacklogマイレポート」）
- **ServiceNow の新店プロジェクト更新通知**
- **ServiceNow の「【未承認の申請があります】」**（承認待ちのリマインダー）
- **ServiceNow の「【承認依頼】」**（例：店舗iPod追加、機器追加依頼 等）
  - ※同じ ServiceNow でも `【通知】`／`【申請が最終承認されました】`／`【差戻】`
    などはスキップせず確認対象に残す。スキップするのは上記の明示パターンのみ。
- **Cisco Security の New Sign-On Notification**（`from:no-reply-security@cisco.com`）
- **Gemini の自動メモ通知**（`from:gemini-notes@google.com`、件名「メモ:「…」」形式の会議メモ自動通知）

## 既読化の手順

- **既読化は必ず本人の指示があってから実行する。Claude の判断で勝手に既読化しない。**
  スキップ対象であっても、提示・確認までにとどめ、既読化は指示を待つこと。
- Claude から既読化できる（`modify_gmail_message_labels` で `UNREAD` ラベルを外す）。
  - 既読化: `remove_label_ids=["UNREAD"]`
  - アーカイブ: `remove_label_ids=["INBOX"]` / ゴミ箱: `add_label_ids=["TRASH"]`
- スキップ対象を一括既読する前に、**カテゴリ別に件名の一覧**を表示して本人の確認を取ること
  （件数だけでなく、どのメールを既読化するか件名まで見せる）。
- 既読化候補の抽出クエリ例: `is:unread in:inbox from:gemini-notes@google.com`
